## 1) Graveyard Principle
Graveyard is active workspace, not a sink.
It should be read as:
- pre-ratchet holding basin
- rescue workspace
- failure-trace library
- classical/nonclassical boundary map

Nothing begins as earned truth.
Whole candidate families may begin in the graveyard basin and only later earn their way out through lower-loop pressure.

## 2) Rescue Use
A1 should use graveyard for:
- rescue transforms
- retry structures
- branch variation against prior fail classes
- explicit recovery campaigns

## 3) Graveyard Library Entry Shape
When the graveyard is used as a structured library, entries should preserve:
- candidate or model family name
- source tradition / source refs
- core insight worth preserving
- classical baggage / residue
- failure class / kill reason
- minimal failing example or branch
- retool attempts
- linked negative-sim / SIM evidence refs
- current status

## 4) Rescue Expectations
Rescue work should preserve:
- family linkage
- source failure reason
- expected new pressure
- SIM relevance

## 4.1) Graveyard Cluster Topology
Graveyard entries may be grouped by shared failure classes.
This topology is a control surface, not ontology.

Use it for:
- graveyard-aware pruning
- negative-sim reuse
- counterexample attachment
- branch-budget steering away from already dense failure basins
- identifying sparse or weakly-tested graveyard regions for new exploration

Do not use it for:
- truth promotion
- replacing lower-loop evidence
- treating cluster structure as earned system law

## 5) Purpose
This file keeps graveyard use structural and operational rather than narrative.
