# FOUNDATION_MODE_AND_SCAFFOLD_MODE_SPLIT__v1
Status: PROPOSED / NONCANONICAL / ACTIVE EXECUTION POLICY
Date: 2026-03-06
Role: Explicit split between machinery-proof work and real foundation-building work

## 1) Purpose
This document prevents the system from mistaking narrow technical proof runs for the main ratchet mode.

The machine needs both:
- scaffold mode
- foundation mode

They are not the same thing and should not be judged by the same success criteria.

## 2) Scaffold Mode
Scaffold mode exists to prove that the machinery works.

Primary goals:
- transport works
- packet/job wiring works
- admissibility path works
- lower-loop ordering works
- one clean executable ladder can survive

Typical shape:
- narrow family
- tighter budgets
- steelman-first or cleaner-first ordering
- smaller graveyard
- thinner negative pressure

Good uses:
- proving a new route exists
- proving a new packet family exists
- proving seeded continuation / rescue / prepack / wrapper behavior
- proving a family-specific SIM lane is reachable

What scaffold mode does not prove by itself:
- rich basin behavior
- broad retool efficacy
- rich graveyard topology
- strong negative-sim pressure
- deep foundation quality

## 3) Foundation Mode
Foundation mode exists to build the deep, rich base the ratchet actually needs.

Primary goals:
- branch rich candidate families from the constraints
- fill graveyard with meaningful structure
- preserve failure topology
- retool outside ideas under the root constraints
- generate richer negative sims and rescue structures
- produce stronger support for later engine / world-model / holodeck work

Typical shape:
- broader batches
- entropy-first
- math-class exploration from the root constraints
- graveyard-first as normal, not exceptional
- richer negative pressure
- richer rescue families
- comparison between multiple retool variants

Good uses:
- entropy-family quarry work
- compatible math-family branching
- classical engine residue stripping
- retooled Bayes / Markov / FEP graveyard exploration
- graveyard cluster growth and failure-basin mapping

## 4) Constraint-First Rule
Foundation mode starts from:
- `F01_FINITUDE`
- `N01_NONCOMMUTATION`

It does not start from imported “fundamentals.”

So:
- density matrices
- probes
- channels
- entropy forms
- engine structures
- Bayes / Markov / FEP retools

are candidate survivors under the constraints, not starting primitives by declaration.

## 5) Current Drift To Correct
Recent work has been too biased toward scaffold mode:
- narrow executable ladders
- local route proving
- technical continuation/debug work

That work was necessary.
It is not enough for the real purpose of the ratchet.

The correction is:
- keep scaffold mode for proving machinery
- shift upper-loop effort back toward foundation mode for real branch-rich, graveyard-rich exploration

## 6) Active Rule
Near-term runnable work may still use scaffold mode where needed.

But upper-loop planning should now prefer:
- foundation-mode batches
- entropy-first branching
- compatible math-class branching
- retool-heavy graveyard growth

unless the immediate blocker is a true machinery failure.
