# Proto-Ratchet Admissible Branch Space

**Date:** 2026-03-29  
**Status:** Proto-ratchet branch-space packet. This file expands the space of still-admissible math and geometry branches downstream of the root constraints without letting the current QIT/Hopf realization masquerade as the only legal endpoint.

---

## 1. Purpose

This packet answers one narrow question:

**Starting from the root constraints, what branch families are still admissible, which branch is currently favored/live, and which things are still too early to close?**

The governing handoff is:

\[
\text{constraints} \to C \to M(C) \to \text{admissible mathematics} \to \text{favored realizations} \to \text{axes}
\]

So this file is about branch space, not closure.

| Root pressure | Immediate effect |
|---|---|

| `N01_NONCOMMUTATION` | order-sensitive composition, noncommuting algebra admitted before geometry |

This root pressure does **not** directly choose one unique geometry.

### 4.1 Strict geometry comparison criteria

Any geometry branch that wants to compete with the current Hopf/Weyl basin should be compared on the same dimensions.

| Criterion | What the rival branch must show |
|---|---|
| root compatibility | it must respect finitude, witness discipline, and noncommutative order sensitivity without smuggling free geometry |
| allowed-math compatibility | it must sit cleanly on the finite admissible packet or another explicitly admitted math basin |
| executable support | it must admit real probes or simulations rather than metaphor-only description |
| carrier richness | it must support state reduction, loop/path structure, and operator action with comparable clarity |
| transport/loop structure | it must reproduce something as informative as the current fiber/base and connection picture, or explain why a different structure is better |
| auxiliary burden | it should not require noticeably more ad hoc imports than the current favored basin |
| anti-smoothing safety | it must not only “feel possible”; it must survive explicit comparison without borrowing strength from the Hopf packet itself |

| Geometry branch read | Status | Why it currently stands there | What would strengthen it | What would kill or weaken it |

| geometry | compatibility geometry on `M(C)`, adjacency geometry, refinement geometry, graph/correlation geometry, information geometry, Hopf-like attractors | `S^3`, Hopf, `S^2`, `T_eta`, fiber/base, Clifford/nested Hopf tori, Weyl working layer |

So:

- the favored live basin is real and well supported
- it is not the whole admissible branch space
- proto-ratcheting should preserve the larger branch space while still ranking the current attractor
